# REF Philosophy

REF treats procurement requirements as engineering objects rather than prose.

A good requirement is not one that sounds professional. A good requirement is one that two independent evaluators can assess consistently using the same evidence.

## Optimization target

REF optimizes for:

1. objective evaluation;
2. supplier clarity;
3. reduced evaluator effort;
4. procurement defensibility;
5. reusability across procurements.

REF does not optimize for long-form explanation, persuasive language or vendor marketing terminology.

## Separation of concerns

REF separates:

- requirement;
- evidence;
- verification;
- scoring;
- rationale;
- contract text;
- implementation guidance.

Combining these concepts in one paragraph creates ambiguity and evaluation risk.

## Configuration over prompting

Configurable choices belong in policies, not in prompts.

Examples:

- maximum scoreable ratio;
- maximum requirement length;
- default evaluation method;
- mandatory keyword;
- language;
- scoring scale;
- accepted evidence types.

## Knowledge, not customer content

REF contains engineering knowledge, not customer-specific requirements.

The repository must not contain:

- supplier names;
- customer names;
- internal procurement documents;
- confidential project details;
- pricing;
- evaluation decisions;
- personal data.

Knowledge packs describe reusable engineering patterns and capability expectations.
