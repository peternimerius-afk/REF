# Evaluation Strategy

Evaluation strategy determines whether a requirement is binary, scoreable or unsuitable for evaluation.

## EV001 Binary by default

The default evaluation method is binary: Yes/No or Pass/Fail.

Binary evaluation is preferred because it minimizes evaluator effort and reduces scoring ambiguity.

## EV002 Score only when justified

A requirement shall be scoreable only when all conditions are true:

1. suppliers can objectively exceed the minimum acceptable level;
2. the difference matters to procurement success;
3. the difference can be measured or consistently assessed;
4. scoring improves supplier selection more than it increases evaluation effort.

## EV003 Split before scoring

If a requirement contains independent obligations, split it before considering scoring.

Do not use scoring to hide poorly decomposed requirements.

## EV004 Minimum first

Where a minimum acceptable level exists, define it as a mandatory binary requirement before defining any scoreable maturity criterion.

## EV005 Maturity may be scoreable

Maturity, resilience, operational excellence, API ecosystem quality, governance maturity and support quality may be scoreable where objective scoring levels can be defined.

## EV006 Compliance is normally binary

Legal, security, privacy and baseline operational controls are normally mandatory binary requirements.

## EV007 Scoreable ratio

The maximum share of scoreable requirements is policy-controlled.

The default maximum is 25%.

## EV008 Scoring scale

The default scoring scale is 1–4.

Each score level shall be defined using objective criteria.

## EV009 Scoring justification

Every scoreable requirement shall include a short justification explaining why scoring creates procurement value.

## EV010 Reject subjective scoring

A requirement shall not be scoreable if the scoring levels rely on subjective judgement without objective criteria.
